Manage Jobs

3.1 Dashboard

The Dashboard page is the default view that opens when you launch Tapdata Replicate , as shown in the following figure. It lists all replications jobs you have defined. You use this view to view,edit,run,and delete jobs,or to create new jobs.

By default it shows all the jobs, including running ones and stopped ones.

To view running jobs only, click "Running" in the toolbar.

To view stopped jobs only, click "Paused" in the toolbar.

3.2 Create Job Page

Click "Create" on the top right to create a new replication job. On the job page, you will be asked to fill out following job details: 

- Enter Job name
- Select sync mode
- Select source connection
- Define model mapping(optional)
- Select target connection

Job Name

Enter a job description

Select sync mode

Choose the way you want to replicate tables to NoSQL database. Currently two options are supported:

  • Database Clone

Choose this mode if you would like to clone your relational database to MongoDB as is. For example, if there are 10 tables in your Oracle database, then after replication there will be 10 collections in MongoDB with exactly same data structure.

  • Custom

Choose this option if you would like to perform model transformation, such as merging data from multiple tables into one JSON collection.

Select source connection

Select the source connection as the replication source. You can create or manage connections on Manage Connections page.

Edit Mapping

When you click mapping, a mapping editor window appears below. The mapping window is used to visually assist you to create a relational table to JSON transformation. See below "Edit Mapping" section for more details on mapping.

Select target connection

Select the target connection as the replication target. You can create or manage connections on Manage Connections page.

3.3 Mapping Editor

The mapping editor is horizontally divided into 3 panes: Properties Pane, Source Pane and Target Pane.

When editor is initialized, it will automatically load the schema from relational database and draw an ER diagram in the Source pane. When there are many tables in the database, only first 20 are displayed. You may use the "Table Filter" in the Properties Pane to select which tables to be displayed in the ER Diagram pane.

3.4 Define a mapping rule

To define a table to collection(Relational to JSON) mapping, follow these steps:

  • Drag a table you would like to replicate data from
  • Move to a blank area in the target pane.
  • Release your mouse.
  • A collection with same name will be automatically created.
  • A mapping line connects the table in the Source Pane to the new collection in the Target Pane is also automatically created.
  • Click on the line to customize the mapping rule properties.

How to define the rule properties

Name

This is descriptive name for the rule

Mapping Type

Here you may choose one of three values: Merged Object, Sub document and Array.

  • Merged Object

Used for a simple one to one mapping. The collection will mirror the table structure.

  • Sub document

Table data will be embedded into JSON as a sub document

  • Array

Table data will be embedded into JSON as a sub array. Each row in the table will become one element in the embedded array.

Array can be used to model a 1:N relationship, for instance, an order may have multiple order line items.

Unique Keys

Specify the unique keys that identify the each row in the table. In most cases, this is same as the primary key columns in the relational table. Tapdata needs this information to find the corresponding JSON in the target collection to perform update operation.

You should provide unique keys, otherwise Tapdata may use all columns as unique keys. Using all columns has performance implications.

Array Matching Keys

This is only applicable when you selected "Array" as mapping type. When there is a change to the source table, Tapdata needs to perform relevant update in the array in target JSON collection. Array matching keys will be used to identify the corresponding array element to update.

Target Path

You may optionally specify the embedded field path name. By default it uses the table name

3.5 Customize Table Properties

You may click on any table to custom certain replication properties related to this table.

Name

Informational only.

Description

Description of the table, informational only.

Custom Sql

Use this text field to provide a custom sql to load the initial data from the table. By default system will use following SQL statement to load the initial data: 

    SELECT * from TABLE_NAME

When using default SQL, system does not track the load progress. When the job is stopped then restarted, it will start from beginning to reload all the initial data.

If you would like to keep track of the load progress and have the ability to resume from the stop point when job is restarted, you need to provide a custom sql with offset tracking column.

For tables with "LAST_UPDATED_TIME" column, you can utilize this type of column to achieve the offset tracking purpose. For example, you may provide a custom SQL as follows: 

    SELECT * FROM ORDERS WHERE LAST_UPDATED_TIME > TO_DATE('${OFFSET1}');

Note the use of ${OFFSET1} . This is a place holder that will be replaced by the specified "Offset" value(provided in below field). Currently only 1 offset column is supported. In the future multiple offset columns may be supported. Hence the suffix number.

Offset

When you use custom sql for offset tracking, you must provide the initial offset value. For example, if LAST_UPDATED_TIME is used as offset column, then you can provide a string value of this column, such as: 

    2018-01-01 12:00:00

3.6 Job Properties

When you click on the gear icon located top right of the screen, or click any blank area within the diagram, the Job Properties view will open.

Sync Type

Inital Sync: Under this mode, Tapdata will only read the current data from source database. The job will stop once all the data is read and replicated to target system.

CDC: Under this mode, Tapdata will continuously read and parse source database's transaction log file and replicate each changes to target system. The job will continue to run until it is stopped.

Initial Sync + CDC: Under this mode, Tapdata will first perform an initial sync. After initial sync Tapdata will then switch to CDC mode to continue replicating changes from source to target.

Sync Point

When Sync Type is CDC, you may specify from which point the CDC operation should start reading data.

BEGINNING: The beginning of the transaction log file.

CURRENT: From the time the job started.

SYNC TIME: You can manually specify the point from which to start change data capture.
